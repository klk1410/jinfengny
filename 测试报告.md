## 四、测试场景

### 4.1 场景说明

> 使用 **5 个小程序进行轮换切换**：当某一小程序出现异常（如限流、风控、不可用等）时，在管理后台切换落地页跳转的目标小程序，商户号则通过第三方平台「易支付」配置支付通道，由小程序内部使用 **JSAPI** 调起微信支付。

### 4.2 主体关系说明

| 链路角色 | 主体归属 |
| --- | --- |
| 落地页域名 | 主体 A |
| 小程序 | 主体 B（共 5 个不同主体） |
| 商户号 | 主体 C（共 4 个不同主体） |

> 落地页、小程序、商户号 **三者主体均不相同**，所有域名均已备案。

## 五、测试结果

### 5.1 测试结论

系统流程正常，但是对接微信部分异常比较多，且不知道详细原因

| 测试项 | 结果 | 备注 |
| --- | :---: | --- |
| 落地页正常访问 | ✅ |  |
| 落地页跳转小程序 | ✅ | 跳转测试正常，小程序异常后需要时间切换 |
| 小程序内拉起微信支付 | ✅ |  |
| 易支付通道转发 | ✅ |  |
| 异常小程序切换 | ✅ | 小程序出现异常后，能够快速自动切换 |
| 多商户号轮询 | ✅ | 在商户功能限制记录后，能够快速切换支付通道 |

### 5.2 异常记录

| 异常现象 | 影响范围 | 处理方式 |
| --- | --- | --- |
| 小程序在提交审核后，无论是否审核通过，都会在1~1.5小时后因为恶意注册封禁 | 落地页跳转小程序；小程序内拉起微信支付 | 切换小程序重新提交审核 |
| 支付通道无法完成支付，在用户微信支付确认阶段，输入密码提交请求到商户的阶段，上报“商户异常，无法完成支付”；但是商户没有功能限制记录，商户有交易拦截记录 | 易支付通道转发 | 由于没有触发警告阈值，手动切换商户 |
